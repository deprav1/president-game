# Архитектура игрового цикла (Game Loop) и состояний

## Problem
Игра изначально была простой чередой карточек, где каждая карточка влияла на 4 параметра. При расширении игры понадобились новые механики:
1. Выборы (каждые 48 месяцев).
2. "Второй шанс" (когда один из параметров достигает 0 или 100).
3. Многоуровневые концовки (зависит от доминирующей фракции).
Смешивать логику карточек и логику специальных экранов в одном потоке стало сложно, особенно учитывая необходимость сохранения стейта в `localStorage`.

## Root Cause
Отсутствие явного конечного автомата (State Machine) для фаз игры. Все события обрабатывались в функции свайпа, что приводило к спагетти-коду при попытке вклинить особые события (выборы, смерть).

## Solution
Введение переменной состояния `phase`. 
Допустимые фазы:
- `onboarding`: Экран приветствия / ввода имени.
- `card`: Обычный геймплей (свайпы).
- `election`: Экран выборов (выбор из 3 тактик).
- `second_chance`: Экран "Второго шанса" перед неминуемой отставкой.
- `gameover`: Экран поражения.
- `victory`: Экран успешного завершения срока.

Функция `choose(side)` теперь работает как диспетчер состояний (Router):
- Если `phase === "second_chance"`, `choose` обрабатывает согласие/отказ от сделки.
- Если `phase === "election"`, `choose` обрабатывает выбор тактики.
- Если `phase === "card"`, `choose` применяет эффекты карты и проверяет условия перехода в другие фазы (через `checkDeath` -> `handleDeathOrRescue` или проверку `months % 48`).

Сохранение в `localStorage` теперь всегда сериализует текущую `phase` и связанные с ней данные (например, `rescueCard`), чтобы игрок мог перезагрузить страницу на любом этапе без потери прогресса.

## Prevention
При добавлении новых "экранов" или механик (например, "Интервью", "Дебаты"):
1. Добавлять новую `phase`.
2. Рендерить соответствующий UI в `App.jsx` на основе `phase`.
3. Добавлять ветку в функцию `choose()` для обработки действий на этом экране.
4. Обязательно добавлять новую логику в объект, который сохраняется в `localStorage`.
